      Main Description

      1. Introduction

      The Software Architecture Design establishes a model that divides the software into components, and defines the interfaces between them as well as the dynamic behaviour of the components together. It creates the basis for detailed development inside each component and ensures that the software is developed in a structured and strategic manner.
      A number of important aspects of the software are (largely) driven by its architecture, such as how reuseable the components are and how easy it is to separate components or plug-in additional ones. Since the architecture defines the separation of software into components, it is created from a high-level overview of the software as a whole.

      2. Building the Architecture

      2.1. Architectural Model

      The layout of the Software Architecture is typically created in the form of a model, in which distinct component blocks are defined, each with its own inputs and outputs and other attributes. Besides the definition of components, this model also captures how the software can be used in multiple variants or product lines. The model is created using UML diagrams. Requirements from the Software Requirement Specification are allocated to each architectural element.

      2.2. Architectural Documentation

      The architectural documentation presents the software architecture in a readable form and captures additional information on the architecture, such as design decisions and rationales. The documentation can be generated from the model, or created based on its information.

      2.3. Integration Test Specification

      Although Integration Testing is not part of the Architecture discipline itself, it as this stage that Integration Tests are also developed for each defined architectural component and the interfaces between them. This ensures that each element of the architecture is covered by integration tests and aids the Software Architect in establishing a design that is both complete, testeable and efficient.

      2.4. Architecture over Sprints

      Since the architectural design is needed by developers before they can start on the Software Detailed Design. Developing the architecture typically takes some time, because it needs to consider a broad view on the complete architecture. Therefor, the Software Architect always works at least one Sprint ahead of the rest of the team. This ensures that the Architectural Model is ready for the components developed in the current Sprint.

      Steps
      Establish Software Architectural Model
      Create the UML model of the Software Architecture, including at minumum the components to be developed during the next Sprint.
      Define and document software interfaces
      Define the interfaces between the software components.
      Define dynamic behaviour
      Define the dynamic behaviour of the identified components (for example scheduling).
      Estimate software resource consumption
      Estimate the software resources that the components will use (RAM, ROM, CPU Load).
      Analyze Software Architecture
      Analyze the software architecture for characteristics such as reuseability, freedom of interference and modularity.
      Review Software Architecture
      Perform Formal Review
      Ensure traceability
      Establish traceability from the Software Requirement Specification to architectural elements and from architectural elements to integration test cases.
