Main Description

Duration of release planning meetings are normally a day. The customer / product owner presents the backlog containing the prioritised Epics to be delivered. Ideally, the Scrum team has already devised rough estimates of how much work is required to implement each of those Epics.

Using the Scrum Team's estimates and the customer's Epic priorities, the team lays out a release plan, mapping Epics very roughly to the first few iterations. Developers may find that fairly low-priority Epics that pose design or architectural risks, and may therefore ask customers to consider assigning them to earlier iterations anyway, in order to address those potential risks as early on in the project as possible.

It is helpful if the development team's velocity on a previous release is already known. In that case, if scope is fixed, the total estimate for all required Epics can be devided by the team's velocity to provide the approximate number of iterations required to deliver the functionality, and thus the deadline. If the deadline is fixed (as is typical), then the velocity needs to be multiplied by the number of iterations to get an initial sense of how many Epics can be delivered. If the development team's velocity is not known, an estimate must be provided for it, and the release plan must be understood to be less precise for the first few iterations, until a reliable velocity number can be derived.
