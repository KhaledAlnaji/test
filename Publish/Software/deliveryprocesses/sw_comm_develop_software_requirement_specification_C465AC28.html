Main Description

1. Introduction

The task to develop a Software Requirement Specification usually starts already before the actual project kick-off. Already during the initial quotation phase of a project, this task is already (although sometimes unofficially or without really recognised as such) running and producing output. Also, it is not a task that has a defined ending. Requirements generally evolve over the course of a project, and thus the task to create and maintan a requirement specification is a continuous facter throughout a project.
Requirements will get added, changed, updated or deleted over time. Additionally, requirements can be vage at the beginning, because not all areas of the project are fully defined yet. Requirements can start at a very high level (for example "The software must have turbo control"), and get more detailed further along the project. So even though in the process workflow this task might look like one that has a defined start and end, in practice it is an ongoing one.
The task to develop a requirement specification can be split into two major elements; Requirement Elicitation and Requirement Analysis. Also here, these elements are not sequential, but cyclic. Throughout the requirement phase, it is common to jump back and forth between these elements.

2. Elicit requirements

Eliciting requirements is the task to gather and clarify requirements. Input requirements will come to a project in many shapes and forms. We bundle them all under one name, Customer Software Requirements, yet this is simply the collection of all incoming inputs to software requirements. This can be a combination of different forms and shapes, such as a detailed specification, models, drawings, presentations or even as knowledge held by a person. Eliciting requirements is the task of distilling real software requirements out of all these different "containers" of requirements.

Naturally, elicitation starts by analysing all incoming documents, models and materials and writing down initial requirements out of these. When the input comes in the form of a detailed software requirement specification already, this might go very fast. The next step is to interview the stakeholders (for example the product owner or technical specialists at the customer) to gather more requirements and/or clarify existing ones. During these talks, a mutual understanding should be reached on the contents of the requirements.
All these requirements are captured in the Software Requirement Specification document.

3. Analyze requirements

After the intial capture of requirements, each documented requirement is analyzed to check if it fulfills all conditions that a requirement should meet. This means ensuring if each requirement is: 
Completeness

Every requirement should be complete. That means that a requirement should have all the information it needs for the developer to understand what exactly has to be implemented. A developer should not need to read other requirements or even documents to find additional information before he can implement the requirement.

  • Consistent (Not contradicting other requirements)
  • Atomic (As small as it can be, not combining multiple requirements into one)
  • Traceable (Its origin is known)
  • Current (Not outdated)
  • Feasible (Can it actually be done or is it an impossible requirement)
  • Unambiguous (A requirement should only be open to one interpretation)
  • Verifiable (Is the requirement testeable)

The analysis is commonly done together with stakeholders and other parts of the team such as architects, testers and developers. If requirements turn out not to meet the conditions, often you will go back to re-eliciting them in order to improve those requirements.
Once all requirements (as far as possible at that stage of the project) are elicited, analyzed and deemed complete, agreement should be gotten from the involved stakeholders. Once that agreement is reached, the requirements can be released for development.

4. Validation Test Specification

Although the Validation Test Specification formally belongs to a different discipline, it gets created at the same time as the Software Requirement Specification does. This ensures that whenever a requirement is created, it is also determined how that requirement will be validated after it has been implemented.
Steps
Collect initial inputs
Collect all incoming documents, models, specifications or other inputs from which software requirements can be distilled.
Perform stakeholder interviews
Perform interviews with all the stakeholders in order to elicit more requirements and clarify existing ones. Document all the results.
Create Software Requirement Specification
Document all elicited requirements in the Software Requirement Specification.
Analyze requirements

Analyze requirements and check if they meet the following criteria:

  • Consistent (Not contradicting other requirements)
  • Atomic (As small as it can be, not combining multiple requirements into one)
  • Traceable (Its origin is known)
  • Current (Not outdated)
  • Feasible (Can it actually be done or is it an impossible requirement)
  • Unambiguous (A requirement should only be open to one interpretation)
  • Verifiable (Is the requirement testeable)
Impact and risk determination
Determine the impact and risk involved with each requirement. Document this in the Software Requirement Specification
Re-elicit requirements
If needed, re-elicit the requirements with the stakeholders to improve or complete them.
Create validation criteria and test cases
Establish for each requirement what its validation criterea are and create (one or more) test cases that will validate the requirement after implementation.
Ensure traceability
Establish and ensure traceability from the input documents and elicitation results to the Software Requirements.
Reach requirement agreement
Reach an agreement on the content of the Software Requirement Specification with all involved stakeholders. Once this is reached, release the requirements for development.
