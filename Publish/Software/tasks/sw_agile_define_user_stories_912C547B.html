Task: Define User Stories
Define User Stories together with the team and the product owner and assign requirements to them.

        Main Description

        1. Introduction

        The project's Product Backlog contains the collection of user stories that together describe the product that will developed. These user stories are the elements on which the Release Planning and Sprint Planning will work.
        A user story is a short description by the Product Owner on a feature of the product. It explains to the team what the product owner expects this feature to do. The product owner (together with the team) will also provide each user story with a priority, which determines its position in the backlog. Like the requirements task, this too is a task that does not have a defined start and end. Typically it re-occurs before each new sprint, so that user stories can be updated and re-prioritized.

        2. Establishing user stories

        User stories are usually created in a meeting with the team and the product owner. It is an open discussion in which the product owner explains his needs and expectation and where the team has the opportunity to ask questions, discuss the stories and clarify the contents of each story. Each story is documented in the product backlog.
        Not every user story needs to be completely detailed from the start. It is very common to have a number of user stories that are quite vague or incomplete, because not all requirements are known yet at that stage. It is important however that user stories with a high priority are given enough detail so the team can start working on them. User stories that are lower on the backlog can stay vague until they start reaching the top. Then they should be further detailed before they need to be planned into a Sprint.
        User stories can sometimes be so vague that they seem very big. In this case they are documented as an "Epic", and designated to be broken up into multiple user stories later on in the project, when more information is available.

        3. Assigning requirements

        Once a user story is sufficiently detailed, requirements can be assigned to it. User stories that are still vague and unclear might not yet have any requirements linked to them, but any requirement that is deemed clear enough to be planned into a sprint should have all relevant requirements for that user story assigned to it.
        At the end of the project, every requirement will have been assigned to a user story.

        Steps
        Define or update user stories
        The team and the product owner together define user stories. The product owner describes his needs and expectation, helped by the team. Each user story is documented in the backlog.
        Prioritize user stories
        Each user story is given a priority, by placing it lower or higher on the backlog
        Assign requirements
        User stories that have enough detail to be assigned to a sprint are assigned all their relevant requirements.
